Question: For an $\FPT$ algorithm, if you lift the requirement that the function of $k$ in the runtime is computable, why is that a problem?

Maybe a simpler question to start with would be this: between any two functions $f(n), g(n): \NN \to \NN$ such that $f(n) = o(g(n))$, is it the case that there exists an uncomputable function in between them? I.e. does there exist a function $h(n)$ such that $h(n) = \Omega(f(n))$ and $h(n) = o(g(n))$? 

Ok, it seems like the answer is largely pedantic, that according to this \href{https://cs.stackexchange.com/questions/152996/why-do-we-care-about-computable-functions-in-parametrized-complexity-definitions}{post}, in the original parameterized algorithms textbook, Downey and Fellows actually address this problem.

\begin{definition}[Fundamental definitions]
	Let $A$ be a parameterized problem
	\begin{enumerate}[(i)]
		\item We say that $A$ is \emph{uniformly fixed-parameter tractable}
			if there exists an algorithm $\Theta$, a constant $c$, and an arbitrary function $f : \NN \to \NN$ such that \begin{enumerate}[(a)]
				\item The running time of $\Theta(\langle x , k \rangle)$ is at most $f(k)|x|^c$, 

				\item $\langle x, k \rangle \in A$ iff $\Theta(\langle x, k \rangle) = 1$
			\end{enumerate}
		\item We say that $A$ is \emph{strongly uniformly fixed-parameter tractable} if $A$ is uniformly fixed-parameter tractable and 
	\end{enumerate}
\end{definition}
